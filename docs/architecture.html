1. System Overview

Super 8s is a Next.js 16 App Router application. Users pick 8 NCAA tournament teams, score seed × wins, and compete on a live leaderboard. The app has two parallel execution paths: the real app (auth-gated, Neon PostgreSQL via Prisma) and a public demo mode (no auth, pure in-memory React context).

Key Schema Design Decisions

  • PlayInSlot decouples play-in uncertainty from picks. Users can pick a "slot" before the play-in game resolves — winnerId is set once the game completes.
  • Pick has dual FK: teamId (direct pick) OR playInSlotId (pick-a-slot). Exactly one is non-null.
  • AppSettings singleton: always id = "main". Use upsert to guarantee existence.
  • Team.wins is the authoritative win count — incremented by ESPN sync, used in all score calculations.
  • TournamentGame.round = 0 means play-in. Rounds 1–6 map to R64 → Championship. Round 0 wins don't count toward user scores.

6. Business Logic: Scoring

All scoring logic lives in src/lib/scoring.ts. The bracket-aware PPR algorithm lives in src/lib/bracket-ppr.ts.

Score (per team): seed × wins (rounds 1–6 only; play-in round 0 = 0)
PPR (potential remaining): seed × max(0, 6 − wins) (alive teams only; 0 if eliminated)
TPS (total projected): currentScore + PPR
Leaderboard sort: TPS desc → currentScore desc → name asc
flowchart TD
    subgraph "computeLeaderboard() — scoring.ts"
        A["Load all users + picks\nfrom Prisma"] --> B["For each user:"]
        B --> C["resolvePickTeam()\nhandle PlayInSlot → Team"]
        C --> D["Sum seed × wins\n= currentScore"]
        D --> E{"Bracket-aware PPR?"}
        E -->|Yes| F["computeBracketAwarePPR()\nbracket-ppr.ts"]
        E -->|No| G["Naive: seed × (6−wins)\nper alive team"]
        F --> H["TPS = currentScore + PPR"]
        G --> H
        H --> I["Sort: TPS → score → name"]
        I --> J["Top 4 get charity preference"]
        J --> K["Return LeaderboardEntry[]"]
    end

Bracket-Aware PPR Algorithm

Naive PPR ignores that two picked teams in the same bracket region can't both win all games. bracket-ppr.ts resolves this with a greedy bottom-up algorithm:

flowchart LR
    subgraph "bracket-ppr.ts: computeBracketAwarePPR()"
        A["Map picked alive teams\nto bracket slots\nseedToSlot(seed) → 0-7"] --> B

        subgraph Intra["Phase 1: Intra-Region R32→E8"]
            B["Group by region\n4 regions × up to 2 teams"]
            B --> C["Walk merge levels\nR32 → S16 → E8"]
            C --> D{"Two teams\ncollide at merge?"}
            D -->|Yes| E["Compare seed × remaining rounds\nHigher PPR team advances\nLoser capped at mergeRound−1"]
            D -->|No| F["Both advance freely"]
            E --> G["Continue to next round"]
            F --> G
        end

        subgraph Cross["Phase 2: Cross-Region F4 (Round 5)"]
            G --> H["East champ vs West champ\nSouth champ vs Midwest champ"]
            H --> I["Same comparison logic\nLoser capped at Round 4"]
        end

        subgraph Final["Phase 3: Championship (Round 6)"]
            I --> J["F4 winners compared\nLoser capped at Round 5"]
        end

        J --> K["Per-team PPR =\nseed × (maxReachableRound − wins)"]
        K --> L["Return {totalPPR, perTeam: Map}"]
    end

7. ESPN Sync Pipeline

sequenceDiagram
    participant Cron as Vercel Cron\n(every 5 min)
    participant API as /api/cron/sync
    participant ESPN as ESPN Public API
    participant DB as Neon DB
    participant Slot as PlayInSlot

    Cron->>API: GET (Authorization: Bearer CRON_SECRET)
    API->>ESPN: fetchESPNScoreboard() [no-cache]
    ESPN-->>API: ESPNScoreboardResponse

    loop For each ESPNEvent
        API->>API: Parse round from competition.notes[0].headline
        API->>API: Extract region (East/West/South/Midwest)
        API->>DB: Upsert Team (espnId unique key)
        API->>DB: Upsert TournamentGame (espnGameId unique key)

        alt Game status = FINAL
            API->>DB: winner.wins += 1
            API->>DB: loser.eliminated = true
            alt Round = 0 (Play-in)
                API->>Slot: PlayInSlot.winnerId = winnerId
            end
        end
    end

    API-->>Cron: SyncResult {gamesUpdated, teamsUpdated, playInResolved, errors}
Round Detection: ESPN embeds the round name in competition.notes[0].headline (e.g. "Elite Eight - East Regional"). The sync function parses this string to map to round numbers 0–6.

Timeline Engine

The game sequence engine derives ~63 individual games from round-level data stored in demo-data.ts:

  • R64 (Round 1): Standard NCAA seed matchups (1v16, 8v9, 5v12, 4v13, 6v11, 3v14, 7v10, 2v15) per region = 32 games
  • R32–E8: Pairs winners by bracket position = 16 + 8 + 4 games
  • F4: East vs West, South vs Midwest = 2 games
  • Championship: 1 game
  • Play-in games: Prepended as Round 0
