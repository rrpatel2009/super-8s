1. System Overview

Super 8s is a Next.js 16 App Router application. Users pick 8 NCAA tournament teams, score seed × wins, and compete on a live leaderboard. The app has two parallel execution paths: the real app (auth-gated, Neon PostgreSQL via Prisma) and a public demo mode (no auth, pure in-memory React context).

Key Schema Design Decisions

  • PlayInSlot decouples play-in uncertainty from picks. Users can pick a "slot" before the play-in game resolves — winnerId is set once the game completes.
  • Pick has dual FK: teamId (direct pick) OR playInSlotId (pick-a-slot). Exactly one is non-null.
  • AppSettings singleton: always id = "main". Use upsert to guarantee existence.
  • Team.wins is the authoritative win count — incremented by ESPN sync, used in all score calculations.
  • TournamentGame.round = 0 means play-in. Rounds 1–6 map to R64 → Championship. Round 0 wins don't count toward user scores.

6. Business Logic: Scoring

All scoring logic lives in src/lib/scoring.ts. The bracket-aware PPR algorithm lives in src/lib/bracket-ppr.ts.

Score (per team): seed × wins (rounds 1–6 only; play-in round 0 = 0)
PPR (potential remaining): seed × max(0, 6 − wins) (alive teams only; 0 if eliminated)
TPS (total projected): currentScore + PPR
Optimal 8: max score achievable by any 8-team combo at current game state (greedy; tie-break by TPS desc)
Leaderboard sort: Points desc → TPS desc → name asc
flowchart TD

        A["Load all users + picks\nfrom Prisma"] --> B["For each user:"]
        B --> C["resolvePickTeam()\nhandle PlayInSlot → Team"]
        C --> D["Sum seed × wins\n= currentScore"]
        D --> E["computeBracketAwarePPR()\nbracket-ppr.ts"]
        E --> H["TPS = currentScore + PPR"]
        H --> I["Sort: Points desc → TPS desc → name asc"]
        I --> J["Top 4 get charity preference"]
        J --> K["Return LeaderboardEntry[]"]
    end
    subgraph "computeOptimal8() — scoring.ts"
        O1["All alive non-eliminated teams"] --> O2["Score each: seed × wins + seed × remaining rounds"]
        O2 --> O3["Greedy: pick top 8 by projected score\n(tie-break: TPS desc)"]
        O3 --> O4["Theoretical ceiling — never decreases"]
    end

Bracket-Aware PPR Algorithm

Naive PPR ignores that two picked teams in the same bracket region can't both win all games. bracket-ppr.ts resolves this with a greedy bottom-up algorithm: 

flowchart LR
    subgraph "bracket-ppr.ts: computeBracketAwarePPR()"
        A["Map picked alive teams\nto bracket slots\nseedToSlot(seed) → 0-7"] --> B

        subgraph Intra["Phase 1: Intra-Region R32→E8"]
            B["Group by region\n4 regions × up to 2 teams"]
            B --> C["Walk merge levels\nR32 → S16 → E8"]
            C --> D{"Two teams\ncollide at merge?"}
            D -->|Yes| E["Compare seed × remaining rounds\nHigher PPR team advances\nLoser capped at mergeRound−1"]
            D -->|No| F["Both advance freely"]
            E --> G["Continue to next round"]
            F --> G
        end

        subgraph Cross["Phase 2: Cross-Region F4 (Round 5)"]
            G --> H["East champ vs West champ\nSouth champ vs Midwest champ"]
            H --> I["Same comparison logic\nLoser capped at Round 4"]
        end

        subgraph Final["Phase 3: Championship (Round 6)"]
            I --> J["F4 winners compared\nLoser capped at Round 5"]
        end

        J --> K["Per-team PPR =\nseed × (maxReachableRound − wins)"]
        K --> L["Return {totalPPR, perTeam: Map}"]
    end

7. ESPN Sync Pipeline

sequenceDiagram
    participant Cron as Vercel Cron\n(every 5 min)
    participant API as /api/cron/sync
    participant ESPN as ESPN Public API
    participant DB as Neon DB
    participant Slot as PlayInSlot

    Cron->>API: GET (Authorization: Bearer CRON_SECRET)
    API->>ESPN: fetchESPNScoreboard() [no-cache]
    ESPN-->>API: ESPNScoreboardResponse

    loop For each ESPNEvent
        API->>API: Parse round from competition.notes[0].headline
        API->>API: Extract region (East/West/South/Midwest)
        API->>DB: Upsert Team (espnId unique key)
        API->>DB: Upsert TournamentGame (espnGameId unique key)

        alt Game status = FINAL
            API->>DB: winner.wins += 1
            API->>DB: loser.eliminated = true
            alt Round = 0 (Play-in)
                API->>Slot: PlayInSlot.winnerId = winnerId
            end
        end
    end

    API-->>Cron: SyncResult {gamesUpdated, teamsUpdated, playInResolved, errors}
Round Detection: ESPN embeds the round name in competition.notes[0].headline (e.g. "Elite Eight - East Regional"). The sync function parses this string to map to round numbers 0–6.

Timeline Engine

The game sequence engine derives ~63 individual games from round-level data stored in demo-data.ts:

  • R64 (Round 1): Standard NCAA seed matchups (1v16, 8v9, 5v12, 4v13, 6v11, 3v14, 7v10, 2v15) per region = 32 games
  • R32–E8: Pairs winners by bracket position = 16 + 8 + 4 games
  • F4: East vs West, South vs Midwest = 2 games
  • Championship: 1 game
  • Play-in games: Prepended as Round 0

AdvancingBracket — Dual Mode

advancing-bracket.tsx is the single bracket component serving two very different contexts:

Mode Key Props Behavior
picks selectedTeamIds, onToggleTeam, disabled Pre-tournament team selection; highlights picked teams; used in PicksForm
simulator gamePicks, onPickGame, gameIndex Click matchups to pick winners; cascade logic propagates picks forward through bracket tree

Cascade algorithm (simulator mode): for each game, participants are derived from prior results — locked games (before gameIndex) feed actual winners forward; unlocked games with a user pick feed that pick forward; missing picks show TBD. This cascades from R64 through Championship.

Leaderboard History Chart

The chart in leaderboard-history-chart.tsx renders three layers simultaneously:

  • Solid lines (0 → gameIndex): actual accumulated scores per user
  • Dashed lines (gameIndex → end): TPS projections using bracket-aware PPR
  • Optimal 8 line: distinct dashed gray reference showing theoretical ceiling — computed in DemoProvider, never decreases as tournament progresses

11. Suggestions & Alternative Architectures

The following are architectural improvements and alternatives worth considering for future sprints or the 2026 tournament cycle.

1. Leaderboard: Move to Real-Time (WebSockets / Server-Sent Events)

Currently the live leaderboard is polled by each client via GET /api/leaderboard with no-store cache. During March Madness, tens of simultaneous users would each trigger a full Prisma query every refresh cycle.

  • Suggested: Use Vercel Streaming or a lightweight SSE endpoint. The cron job writes the computed leaderboard to a KV store (e.g. Vercel KV / Upstash Redis); the SSE endpoint streams the cached result to clients.
  • Benefit: One Prisma query per cron tick (every 5 min) instead of N queries per N clients.
  • Trade-off: Adds Upstash/Redis dependency; slight staleness (≤5 min) acceptable for a contest app.

2. Demo Mode: Pre-Compute at Build Time

The demo recomputes all 63 leaderboard snapshots on every render in DemoProvider. This is intentional for smooth scrubbing but adds CPU cost proportional to user count.

  • Suggested: Generate the full snapshot array as a static JSON file at build time (a prebuild script) and import it directly. DemoProvider would just index into a pre-built array.
  • Benefit: Zero runtime computation; instant scrubbing even with 50+ demo users.
  • Trade-off: Need to regenerate JSON when demo-data.ts or demo-users changes. Easy to automate.

3. Bracket-Aware PPR: Memoization

computeBracketAwarePPR() is called once per user per leaderboard refresh, and re-runs from scratch each time. With 12 demo users × 63 snapshots, this compounds quickly.

  • Suggested: Cache results keyed by (pickedTeamIds[], teamStateHash) using a Map or useMemo in the context. Team state only changes when gameIndex changes.
  • Benefit: Near-zero cost on slides where team state hasn't changed (e.g. scrubbing within a round).

4. Scoring: Move computeLeaderboard to a Server Action

The leaderboard page currently directly queries Prisma inside an RSC, which is fine but tightly couples the page to the DB. There's also a separate /api/leaderboard route that does the same computation.

  • Suggested: Canonicalize to a single Next.js Server Action or a shared getLeaderboard() server function. Remove the duplicate API route if the leaderboard page is the only consumer.
  • Benefit: Single source of truth, easier to add caching via unstable_cache.

5. Auth: Add an Invite-Only Guard

Currently any email can sign up. For a private contest with a fixed set of participants, an allowlist would prevent uninvited users from creating accounts.

  • Suggested: In the signIn callback in auth.ts, check the email against AppSettings.allowedEmails (a new JSON field) or a separate Allowlist model. Return false to reject uninvited signups.
  • Benefit: No need to manually manage isPaid flags or bulk-delete accounts.

6. Demo Mode: Multi-Year Dataset as a DB Table

Right now adding a new tournament year means creating a new demo-data-YYYY.ts TypeScript file and a demo-users-YYYY.ts file. This is fine for 1-2 years but grows unwieldy.

  • Suggested: Store tournament data in a read-only TournamentSnapshot table (year, teamsJson, usersJson). The demo engine reads from DB on initial load instead of from static files. Admin UI can upload new tournament CSVs.
  • Benefit: Admins can add historical years without a code deploy.
  • Trade-off: Demo mode is no longer zero-DB — acceptable given it's read-only and cacheable.

7. Bracket Components: Unify bracket-view.tsx and advancing-bracket.tsx

There are now two interactive bracket components: bracket-view.tsx (picks page, per-region tabs) and advancing-bracket.tsx (full 64-team, simulator/picks dual-mode). Their feature sets overlap significantly.

  • Suggested: Consolidate into advancing-bracket.tsx as the single system-wide bracket primitive. Add a layout="tabs" prop to toggle the per-region tab view used on the picks page. Deprecate bracket-view.tsx.
  • Benefit: One place for conflict detection logic, one CSS layout system, one set of tests.

8. ESPN Sync: Dead-Letter Queue for Failed Games

If the ESPN API returns an error mid-sync, syncTournamentData() currently logs the error and returns a partial SyncResult. Failed game upserts are silently dropped.

  • Suggested: Persist failed ESPN game IDs to a SyncError table. On the next cron tick, retry failed items first. Surface pending errors in the Admin Sync page.
  • Benefit: Guarantees eventual consistency even during transient ESPN API hiccups during the tournament.
